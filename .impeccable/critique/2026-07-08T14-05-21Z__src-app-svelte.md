## Anti-Patterns Verdict

**LLM assessment:** Mostly clean and bespoke — no gradient text, no hero-metric tiles, no glassmorphism, no numbered section markers, no colored side-stripe borders (verified in `Card.svelte`/`AssetCard.svelte`), both themes are real token remaps not filters. The inline `{doc}`/`{n}` variable-fill pattern is genuinely bespoke design, not template output. However, two concrete self-contradictions of the app's *own* anti-pattern list were found: `.asset-card__panel-label` ("FILL IN AND COPY") is the literal uppercase-eyebrow-label pattern `PRODUCT.md` names as an anti-reference, and a long unbroken token overflows its card into the neighboring column in grid view (the "text overflow" tell).

**Deterministic scan:** Static CLI scan (`detect.mjs --json src`) returned clean — exit 0, zero findings across 47 scannable files. This is a **false negative** relative to the LLM's manual finding: the eyebrow-label markup exists in source but the static ruleset didn't flag it. The **runtime/browser-injected** detector (a different, computed-style-based pass) found what the LLM review did not: two **low-contrast** violations in dark theme — white text (`#ffffff`) on Indigo Soft (`#a5adfb`) at **2.1:1** (need 4.5:1) on the "New asset" button and the active filter chip — plus a **skipped heading level** (`<h1>` → `<h3>`, no `<h2>`) and a flat type-scale observation (12–20px, ratio ~1.7:1, consistent with the system's deliberate "No-Hero Rule" so likely not actionable on its own).

**Visual overlays:** Browser injection succeeded; the on-page overlay rendered the flat-type-hierarchy banner live, and the structured API surfaced the contrast and heading findings above. No overlay remains open — the live-server instance used for detection was stopped after evidence gathering.

## Overall Impression

The system is disciplined and mostly walks its own talk — the inline fill-and-copy interaction is a genuinely strong, bespoke idea, and the token-driven light/dark system is well-executed. But it has real gaps between what `PRODUCT.md`/`DESIGN.md` promise and what's shipped: a dark-theme contrast failure that violates the stated WCAG AA baseline, zero keyboard-driven interaction despite an explicit Raycast/Linear positioning, and a status color (amber) doing double duty as both "Tool" badge and "favorited" — undermining the system's own "One Signal Rule." The biggest opportunity is closing that gap between stated principles and shipped behavior, not a redesign.

## What's Working

1. **The inline variable-fill pattern** — typing directly into `{doc}`/`{n}` slots inside the prompt text itself makes the content the interface, exactly per the "Content is the interface" design principle.
2. **Import-as-diff.** Computing `toAdd`/`conflicts` and requiring per-conflict resolution before any mutation is real error prevention, not just a confirm dialog.
3. **Dark theme as a genuine token remap** — every component reads semantic `--color-*` roles with zero theme-branching in component code.

## Priority Issues

- **[P0] Low-contrast text on accent surfaces in dark theme**
  **Why it matters:** White text (`#ffffff`) on Indigo Soft (`#a5adfb`) measures 2.1:1 on the primary "New asset" button and the active filter chip — a hard WCAG AA failure (need 4.5:1) on two of the most-used controls in the app, directly contradicting `PRODUCT.md`'s stated accessibility baseline.
  **Fix:** In the dark theme, either darken the accent used behind white text (closer to `brand-500`/`brand-600`) or switch `--color-text-on-brand` to a dark ink color when the background is the lighter `brand-300` step.
  **Suggested command:** /impeccable audit

- **[P0] Text overflow breaks card layout in grid view**
  **Why it matters:** An unbroken long token (URL, hash, file path) bleeds out of its card into the neighboring column — a plausible real-world case for prompt content, and literally the "text overflow" AI-slop tell this system otherwise avoids.
  **Fix:** Add `overflow-wrap: anywhere` to `.asset-card__preview`/`.asset-card__fill` and `overflow: hidden` on the card as a backstop.
  **Suggested command:** /impeccable harden

- **[P1] No keyboard-driven path, contradicting the Raycast/Linear promise**
  **Why it matters:** `PRODUCT.md` explicitly invokes Raycast/Linear-grade speed, and the job-to-be-done is "get in, get the asset, copy it, get out in seconds" — but there's no shortcut to focus search, open "New asset," or navigate cards without a mouse.
  **Fix:** At minimum, a global `/` or `Cmd+K` to focus search and `Cmd+N` for New Asset.
  **Suggested command:** /impeccable shape

- **[P1] Amber (status color) double-duty on the favorite star**
  **Why it matters:** The favorite toggle's active state reuses `--color-warning`, the same amber assigned to the "Tool" type badge — violating `DESIGN.md`'s own Named Rule that status colors are never decorative or reused, and diluting the "One Signal Rule" precision the system is built around.
  **Fix:** Give favorite its own token, or better, use the Electric Indigo accent since favoriting is an actionable/selected state.
  **Suggested command:** /impeccable colorize

- **[P2] Inconsistent destructive-confirmation pattern**
  **Why it matters:** Delete swaps one modal for another; Reset stacks a second modal on top of the first with double-dimmed backdrops and no stack-awareness in `Modal.svelte`'s per-instance keydown listener — a concrete keyboard-trap risk for screen-reader/keyboard-only users at exactly the moment reassurance matters most.
  **Fix:** Standardize on one pattern (replace parent modal content rather than stacking).
  **Suggested command:** /impeccable polish

- **[P2] Skipped heading level**
  **Why it matters:** The page structure goes `<h1>` → `<h3>` with no `<h2>`, breaking the landmark navigation screen-reader users rely on.
  **Fix:** Insert the missing `<h2>` level or renumber the existing headings to be sequential.
  **Suggested command:** /impeccable audit

- **[P3] Confirmation copy degrades with long titles**
  **Why it matters:** "Delete `<full title>`? This can't be undone." buries the actual question in a wall of bolded text when the title is long.
  **Fix:** Truncate the interpolated title (~60 characters with an ellipsis).
  **Suggested command:** /impeccable clarify

## Persona Red Flags

**Alex (Impatient Power User):** No keyboard shortcut to jump into search (the #1 friction point). More seriously — the header's "Copy to clipboard" icon on a *collapsed* card copies the raw prompt with literal `{doc}`/`{n}` braces still in it; only the expanded panel's "Copy" produces the filled/resolved version, and nothing signals the difference. Moving fast, Alex is more likely to hit the wrong one and paste broken output.

**Sam (Accessibility-Dependent User):** The stacked-modal scenario (Data → Reset) is a concrete, code-verifiable keyboard-trap risk: each `Modal.svelte` instance attaches its own global keydown listener with no stack-awareness, so Escape/Tab behavior is unpredictable with two modals open. Icon-only header controls have correct aria-labels for screen readers but no visible hover tooltip for low-vision users navigating visually.

**Riley (Deliberate Stress Tester):** Confirmed all three visual bugs firsthand — the grid-view text overflow, the delete-confirmation copy degradation with a long title, and the stacked-modal backdrop-doubling on Reset. Rapid re-click of "Copy" correctly reset the 2-second "Copied" timeout with no bug found.

## Minor Observations

- Empty-state and no-results views leave a large, unbalanced dead zone below a small centered box.
- The favorite star uses Unicode glyphs (★/☆) while every other icon is custom SVG.
- In dark theme, the New Asset form's Roles/Slots fieldset hairline borders are nearly invisible, weakening the grouping cue.
- `Modal.svelte` scrolls its header and footer along with the body — a long form can scroll the title and primary action out of view together.
- Filter chips stay fully populated even when a search yields zero matches — a missed status-visibility opportunity.
- Flat type hierarchy (12–20px, ratio ~1.7:1) flagged by the runtime detector is consistent with the system's deliberate "No-Hero Rule" and is likely not actionable on its own.

## Questions to Consider

- The brand explicitly cites Raycast and Linear — both keyboard-first products. Does a fully mouse-driven interface actually earn that comparison, or does "The Quiet Console" need a command-palette moment?
- Category (13 options) and Roles (7 options) both exceed the "≤4 visible options" working-memory guideline — is this taxonomy serving retrieval speed, or has it grown past what a single fast-moving user needs?
- With no undo and no typed-confirmation step, is a plain confirm dialog enough protection for Delete/Reset given the "local-first trust" principle this system is built on?
